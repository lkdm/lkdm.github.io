Build and automate your toolbox

2025-02-24

For years I've kept my machines somewhat in sync with a sparse group of dotfiles in a git repository. I have been doing what most people do, and writing a setup script to symlink them to the correct spots.

This works somewhat well, but it necessitates that I manually symlink the dotfiles to the correct locations.

The problem with this setup started when I was unceremoniously handed a laptop with Windows pre-installed instead of a Mac at my most recent role. I can't stand Windows, and besides have very little experience trying to shoehown a POSIX development environment onto it, so I was determined to give Linux a try.

Luckily my employers were understanding and supportive, and so I booted up Ubuntu and pulled down my dotfiles.

After some teething issues, I fixed a bunch of things with my configuration. But this caused a few issues;

  1. The packages I had installed on my personal Mac and my work computer were different.
  2. The dotfiles sparsely covered the system, giving a lot of room for problems.
  3. Differences in shell and system led to problems.

Why you absolutely should automate your toolbox

To master your tools

If you are a software engineer, you are likely to be doing this job for a long time. Conservatively, years, but likely for decades.

Learning the same set of tools really, really well is an investment into yourself.

To keep context switching minimal

I love programming, and sometimes race home from work with a great idea that I just need to code. Having a personal dev environment that is radically different than my work dev environment was a show stopper every time.

Be productive, even in the face of disruptions

Our physical reality is unpredictable. You should backup your data, and this includes the state of your machine. You should expect disruptions, and you should be able to have a productive day even if your computer dies and you need to quickly use another one.

Quick selection of tools

Here is my reasoning for some of the tools I've chosen:

Zsh

I've chosen zsh as a shell because it has some great features, and comes default on Mac OS systems. I use mine without oh-my-zsh, which although popular and great, slows down the response time in the terminal too much for my liking. It should be snappy!

How I automated my toolbox

Use a dotfiles git repository

If you haven't already, create a remote git repository (on a Git platform such as GitLab, BitBucket or GitHub), call it dotfiles or something similar, and clone it to your computer.

git clone git@github.com:USER/dotfiles.git

I like to mirror the git URL on my computer:

cd ~/github.com/USER/dotfiles/

Then create a symbolic link

ln -s ~/github.com/USER/dotfiles/.zshrc ~/.zshrc

While this is all you technically need to get started with dotfiles, we can do a lot better.

Use a password manager for secrets

Install nerdfonts

The first thing I used to do when setting up any new system was go and install a NerdFont, this was always mildly painful because in addition to downloading the font, I'd have to figure out where to put the file, then hope that past-me remembered to put the configuration files that reference them on git.

Chezmoi lets you write a script that automatically installs them to the correct location.

Use atuin

Some developers actually sync their .bash_history file.

I personally use atuin which is a cli utility that stores shell history in a SQLite file. It can also sync to an encrypted db on the cloud, for free.

I love it. Now that my entire config is in sync, syncing my shell history helps me rediscover shell commands that I've used to complete various things.

I also have a dotfile for atuin that keeps the configuration in sync.

Use a script to install packages

I still haven't arrived on a great way to keep packages synced across my systems.

At the moment I've settled on this behemoth of a script that lets me declare packages in different managers across systems.

I'm also looking into the feasability of using Homebrew's Brewfiles.

I've even looked into using Nix and devenv.sh for package management, but decided against is because I feel like learning these tools would be like a second job. My current setup just works.
